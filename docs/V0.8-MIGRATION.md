# HIVE v0.8 — Unified Migration Plan

> Status: **PLAN** (not started). One coordinated breaking change with a
> single production hard reset. Supersedes the piecemeal v0.7.8/.9 items.
> Owner: capy. Drafted 2026-05-27.

---

## 1. Why v0.8 (the problems we're solving)

The v0.7.x architecture has the queen do **all** vectorization. Every
fragment from every bee passes through the queen's Python embedder. With
1–2 bees this already caused the recurring incidents of this cycle (GIL
contention → embedder OOM/timeouts, replication stalls). With hundreds of
bees it is structurally impossible — the queen is a hard bottleneck.

Three coupled limitations, fixed together so there is **one** reset, not three:

1. **Embedding model** — `all-MiniLM-L6-v2` (384-d, English-centric) is the
   root of the cross-lingual gap (Spanish query vs English corpus) and the
   weak retrieval precision. → Switch model = re-embed everything (qdrant reset).
2. **Where vectorization happens** — moving it to the bee (producer-side) is
   HIVE's own thesis (*distributed production*). The fragment's Hypercore
   format must change to carry the vector. → New core format.
3. **The vector store + runtime** — Python embedder + Qdrant (a separate
   service) → an **all-Node** stack (embeddings via ONNX, vectors in LanceDB)
   kills the Python GIL class of bugs and simplifies deploy. → Index migration.

Doing these separately = 3 disruptions + 3 resets. The network is young
(1 bee, data re-crawlable) so **one coordinated cutover** is cheaper and cleaner.

## 2. End-state architecture

```
BEE (producer, ~600–900 MB)                 QUEEN (thin aggregator, light)
 ├─ ForagerSource extract (verbatim)         ├─ replicate bee Hypercores (P2P)
 ├─ deterministic layout chunking            ├─ READ vectors from fragments
 ├─ embed each chunk  (e5-base, ONNX int8)   │   (NO embedding work)
 ├─ sign {text+meta+vector} (ed25519)        ├─ upsert into LanceDB (embedded)
 └─ append to its Hypercore (vector inline)  ├─ embed only the QUERY (e5 "query:")
                                             └─ LanceDB ANN + filters → LLM synth
```

Key shift: **the queen never embeds passages.** It copies the bee's
pre-computed, signed vectors straight into the index. A queen can therefore
aggregate hundreds of bees because its per-fragment cost is an upsert, not a
transformer forward pass.

### Standard model (network-wide invariant)

All bees + the queen MUST use the same embedding model+version, or vectors
are not comparable in one index. Locked for v0.8:

- **Model:** `intfloat/multilingual-e5-base`
- **Dim:** 768
- **Runtime:** ONNX, int8-quantized weights, via `@huggingface/transformers`
  (transformers.js). Optimise to raw `onnxruntime-node` later if throughput needs it.
- **Prefixes (e5 requirement):** passages embedded as `"passage: <text>"`,
  queries as `"query: <text>"`.
- Declared in `BeeManifest.embedding_model` / `embedding_dim`. The queen
  rejects (or segregates) vectors whose model/dim don't match the standard.

Changing the model later = network-wide version bump + reset. Chosen now
deliberately for quality + multilingual coverage.

## 3. Fragment v0.8 schema (FINAL)

Universal across all sources, signed (text + metadata + vector), versioned.

```ts
interface Fragment {
  // A. Identity & version
  id: string;
  schema_version: number;          // 2 for v0.8 (1 = legacy v0.7). Enables clean future evolution.
  node_id: string;                 // producing bee
  node_pubkey: string;             // ed25519 pubkey → self-verifiable without PeerRegistry

  // B. Content
  text: string;                    // verbatim
  lang: string;                    // BCP-47 (es, en, …) — per fragment
  title?: string;
  content_hash: string;            // SHA-256( normalize(text) ) — see §6.3. Enables corroboration.

  // C. Provenance (Dublin Core / schema.org flavour, source-agnostic)
  source: string;                  // adapter id: "wikipedia-en"
  source_type: string;             // "wikipedia" | "arxiv" | "rss" | "commoncrawl" | "custom"
  url: string;                     // canonical URL
  license?: string;                // SPDX-ish ("CC-BY-SA-4.0", …) — license propagation for training corpora
  identifiers?: Record<string,string>; // { doi, arxiv, pmid, isbn, … } — replaces doi/arxiv_id, extensible
  retrieved_at: string;            // when the source was fetched

  // D. Structure / chunking
  section_path?: string[];         // ["Physiology","Photosynthesis"]
  chunk_index?: number;
  chunk_count?: number;

  // E. Embedding (bee-side)
  vector: string;                  // base64(Float16Array, 768) — see §6.1
  embedding_model: string;         // "intfloat/multilingual-e5-base"
  embedding_dim: number;           // 768

  // F. Lifecycle / TTL
  extracted_at: string;
  ttl_seconds?: number;            // freshness window by source (wiki 7d, rss 24h, arxiv 30d). expires_at is derived.
  status: 'current' | 'superseded' | 'historical';
  supersedes: string[];
  superseded_by: string | null;

  // G. Coordination (v0.7.6)
  partition?: string;              // scope partition that produced it

  // H. Integrity (signed)
  confidence: number;
  hash: string;                    // SHA-256 of the canonical payload INCLUDING the vector
  signature: string;               // ed25519 over { id, hash }
}
```

**Removed from v0.7:** `doi`, `arxiv_id`, `doi_valid`, `vector_id`
(→ `identifiers` + inline `vector`).

**Queen-side / derived (NOT in the fragment, NOT signed):**
`corroboration_count` (computed by counting equal `content_hash` across
distinct bees), per-query `score`, `expires_at` (= extracted_at + ttl).

### BeeManifest additions
```ts
embedding_model: string;   // must equal the network standard
embedding_dim: number;
chunker_version: string;   // deterministic-chunking algo version (corroboration needs identical chunking)
schema_version: number;
```

## 4. Component changes

| Area | v0.7 (now) | v0.8 (target) |
|---|---|---|
| Embedder runtime | Python `sentence-transformers` | Node `@huggingface/transformers` (ONNX) |
| Model | all-MiniLM-L6-v2 (384-d, EN) | multilingual-e5-base (768-d, int8) |
| Who embeds passages | the queen | **the bee** |
| Who embeds the query | the queen | the queen (only the query) |
| Vector store | Qdrant (separate container) | **LanceDB** (embedded, in-process) |
| Vector in Hypercore | no (`vector_id` ref only) | **yes, signed, base64 fp16** |
| Chunking | section + char threshold | **deterministic layout-based** |
| Python in the stack | yes (embedder) | **none** |

## 5. Migration sequence — build incrementally, cut over once

Develop and validate each phase in a branch; production stays on v0.7.7.x
the whole time. Only Phase 5 touches prod, as a single reset.

### Phase 0 — Branch + scaffolding
- `git checkout -b v0.8`
- Add `packages/embeddings-node` (or replace `packages/embeddings`).
- Pin model + dim + prefixes as shared constants.

### Phase 1 — Node embedder (e5-base, ONNX int8) — *isolated* ✅ SPIKE DONE (2026-05-27)
- Load `intfloat/multilingual-e5-base` ONNX int8 via transformers.js.
- `embedPassage(text) → Float32Array(768)`, `embedQuery(text) → …` (e5 prefixes).
- **Validate:** dims = 768; cosine sanity (synonyms close, unrelated far);
  multilingual check ("fotosíntesis" ≈ "photosynthesis"); RAM on target bee
  hardware (target < 900 MB resident incl. Node base).

**Spike results** (`packages/embeddings-node/spike.mjs`):
- ✅ `Xenova/multilingual-e5-base`, `dtype: 'q8'` (int8) loads in ~8 s,
  **RSS ~780 MB**, dim **768**, ~38 ms/embed. Resolves the top risk
  (model exists in ONNX int8).
- ✅ Cross-lingual works: ES query "¿qué es la fotosíntesis?" vs EN
  photosynthesis passage = **0.814**, above the unrelated cocido (0.728).
  The v0.7 cross-lingual gap is fixed.
- ✅ fp16 vector ≈ 0.72 GB extra per 500 k fragments.
- ⚠️ **FINDING — recalibrate the retrieval gate.** e5 cosine scores
  compress to ~0.70–0.91 (noise floor ~0.70, not ~0.25 like MiniLM):
  e.g. EN-query vs unrelated mitochondria still scores 0.819. The
  `RELEVANT_SCORE = 0.45` from v0.7.7 is MiniLM-calibrated and would pass
  everything under e5. **Re-tune in Phase 4** (likely ~0.82–0.86, set
  empirically against a labelled sample); the gating *logic* (score AND
  keyword, LLM grounded-verdict) stays, only the threshold changes.
  Note the slight same-language bias (cross-lingual match sits just below
  same-language topical matches) — the LLM grounded-verdict backstops it.

### Phase 2 — LanceDB on the queen — *isolated DB swap*
- Replace Qdrant client with LanceDB (`@lancedb/lancedb`). Table = id + vector(768)
  + filterable payload (lang, source, source_type, node_id, content_hash, …).
- Keep queen-side embedding for THIS phase (isolate the DB change from the
  embedding change).
- **Validate:** ANN search quality matches/beats Qdrant on a sample set;
  filters work (lang, source); disk + RAM footprint; quantization config.

### Phase 3 — v0.8 schema + deterministic chunking + bee-side embedding
- Implement the Fragment v0.8 schema (§3); bump `schema_version = 2`.
- Deterministic layout-based chunker (`chunker_version`), identical output
  across bees for the same input.
- Bee embeds each chunk (passage prefix) → fp16 base64 → sign whole payload
  (incl. vector) → append.
- **Validate:** two bees, same article → identical `content_hash` per chunk
  (corroboration works); signature verifies with vector in payload;
  Hypercore size per fragment ≈ text + ~1.5 KB vector.

### Phase 4 — Thin queen (consume bee vectors)
- Queen reads `vector` + `embedding_model` from replicated fragments; if model
  matches the standard, upsert the vector into LanceDB **without embedding**;
  else skip/segregate + log.
- Query path: queen embeds only the question (query prefix) → LanceDB search +
  the v0.7.7 retrieval gating + LLM grounded-verdict (keep those — they're
  model-agnostic).
- **Validate end-to-end:** bee extract+embed → P2P replicate → queen upsert →
  `/api/query` returns correct, grounded answers; queen CPU stays flat under
  load (no transformer forward passes).

### Phase 5 — Single production cutover (the one hard reset)
1. Snapshot current prod (qdrant volume + note git tag `v0.7.7.x`) for rollback.
2. Deploy the v0.8 image (Node embedder, LanceDB, new schema). Update
   docker-compose (drop the qdrant service + the Python embedder; add LanceDB
   volume; `stop_grace_period` already in place from v0.7.7.12).
3. Bees start **fresh cores** (vectors inline). Old text-only data is
   re-crawled (Wikipedia is still there) — acceptable for a young network.
4. Queen starts with an empty LanceDB and fills from the bees newest-first
   (fold in the v0.8 newest-first indexing — reverse backfill + live tail).
5. capybarahome `/hive` + README updated per the docs↔version operating rule.
- **Validate:** version badge shows v0.8; queries return fresh + correct;
  RAM/disk within budget; no Python process in any container.

## 6. Resolved design decisions

1. **Vector encoding:** store `vector` as **base64(Float16Array)** in the
   JSON value (~2 KB vs ~6 KB for a raw JSON float array). fp16 ≈ no retrieval
   quality loss. LanceDB applies its own quantization at index time.
2. **Vector is in the signed hash** — cryptographically attributable; a bee
   cannot ship good text + garbage vector deniably. Re-vectorizing = re-sign.
3. **`content_hash` normalization:** `SHA-256( NFC → trim → collapse internal
   whitespace )`, **no lowercasing** (preserve verbatim). Requires the
   **deterministic chunker** so two bees produce identical chunks → identical
   `content_hash` → corroboration.
4. **Derived/queen-side (unsigned):** `corroboration_count`, per-query `score`,
   `expires_at`.

## 7. Risks & mitigations

| Risk | Mitigation |
|---|---|
| e5-base ONNX int8 unavailable / poor in transformers.js | Verify the ONNX export exists (Xenova/HF hub); fall back to fp16 ONNX or self-convert. Spike in Phase 1. |
| Bee RAM too high on small hardware | int8 keeps e5-base ≈ 600–900 MB; offer e5-**small** as a lighter network profile if Pi-class ubiquity matters (separate standard — not mixable). |
| LanceDB scale/filter gaps vs Qdrant | Spike in Phase 2 with a realistic corpus before committing. |
| Chunking not deterministic across bees | Pin `chunker_version`; test two bees → identical content_hash in Phase 3. |
| Vectors bloat the Hypercore (~+1.5 GB/500k) | fp16; accept; selective replication (queen pulls only relevant cores) is a later lever. |
| Cutover breaks prod | Phase 5 takes a qdrant snapshot + keeps the v0.7.7.x image tag → roll back by redeploying the old tag. |
| Re-crawl cost of fresh cores | Young network, data re-crawlable; bees refill in hours. |

## 8. Out of scope for v0.8 (later)
- MCP server + OpenClaw connector (independent of this migration — can ship in parallel).
- Selective replication (Bloom/centroid routing) — a scaling lever once multi-bee is real.
- Reputation / bounties / swarm visualizer — marketing/scale.
- Multi-bee validation in prod is a PREREQUISITE the migration assumes; stand up 2–3 real bees during Phase 4.
